# 第十四章 · 应用范式(RAG / Agent)

> 到这里,模型本身已经讲透了:从注意力、Transformer,到训练、采样、微调、对齐、推理优化。
> 但光有模型不够——它知识停在训练那一刻,也不会算术、不会上网、不会调 API。
>
> 这一章讲两个把模型「用起来」的主流范式:**RAG**——回答前先查资料,让模型用上新知识、少胡说;
> **Agent**——给模型配工具,让它在「想-做-看」的循环里自己分步解决问题。两个都手写最小版,都能跑。

---

## 1. 模型的两个天生短板

前面训出来的模型很强,但有两件事它做不到:

- **知识是「冻结」的**:训练完成那一刻起,新发生的事、你公司的私有文档,它一概不知道。
- **只会生成文字**:它不会做精确算术、连不上网、调不了数据库——纯靠记忆硬答还容易**胡说(幻觉)**。

这一章的两个范式,正好各补一个短板:RAG 补「知识」,Agent 补「动手能力」。

---

## 2. RAG:先查资料,再回答

RAG(Retrieval-Augmented Generation,检索增强生成)的主意很直接:
**回答之前,先去资料库查最相关的几条,塞进提示词,再让模型照着答**。就像开卷考试——先翻到相关那页,再下笔。

一套最小 RAG 就两步:

1. **检索**:把问题和每条资料都变成向量,找最相似的几条;
2. **增强生成**:把查到的资料拼进提示词,交给模型。

检索的核心是「向量化 + 找最近」。这里用最朴素的词袋向量 + 余弦相似度抓住本质:

```python
def retrieve(question, k=2):
    qv = embed(question)                      # 问题变成向量
    sims = [cosine(qv, dv) for dv in doc_vecs]  # 和每条资料算相似度
    top = np.argsort(sims)[::-1][:k]          # 取最像的 k 条
    return [(i, sims[i]) for i in top]
```

> 代码见 [`code/minimal_rag.py`](code/minimal_rag.py)

给一个模型**绝不可能见过**的私有知识库(公司年会、报销流程、WiFi 密码……),提问看看:

```text
问题:WiFi 密码是多少?
  检索到最相关的资料(按相似度):
    相似度 0.440 | 公司 WiFi 名称是 Office_5G,密码是 welcome2024,访客网络无需密码。
    相似度 0.075 | 我们公司的年会定在 12 月 20 日晚上六点,地点是滨江大酒店三楼宴会厅。
  -> 喂给模型后回答:(模型据此作答)- 公司 WiFi 名称是 Office_5G,密码是 welcome2024…

问题:怎么请年假?
  检索到最相关的资料(按相似度):
    相似度 0.354 | 请年假需提前三天在 OA 提交,连休超过五天的需要总监审批。
    相似度 0.105 | 我们公司的年会定在 12 月 20 日晚上六点,地点是滨江大酒店三楼宴会厅。
  -> 喂给模型后回答:(模型据此作答)- 请年假需提前三天在 OA 提交…
```

每个问题都**准确检索到了对应那条资料**(相似度最高),模型的回答因此有了出处——
不是凭记忆编的,而是从查到的资料里来的。这就是「让 AI 读你的 PDF / 知识库再问答」类产品的底层。

> 本例用词袋向量演示;真实 RAG 用神经网络做的**嵌入(embedding)** + 向量数据库,
> 但「向量化 + 找最近邻」的思想一模一样。

## 3. Agent:给模型配工具,让它自己动手

RAG 让模型「会查资料」。但有些任务光查不够,需要**动手**:算一道题、查实时天气、调一个 API。
模型本身干不了这些,于是给它配几样**工具**,让它自己决定何时用、用哪个。

这套循环常叫 **ReAct(Reason + Act,边想边做)**:

```text
想(Thought) -> 做(Action:调某工具) -> 看(Observation:工具返回) -> 再想…… -> 最终答案
```

循环骨架很简单:大脑决定下一步动作 → 调用工具 → 把结果喂回去 → 再决定,直到给出答案。

```python
def run_agent(question):
    history = []
    for step in range(max_steps):
        decision = brain(question, history)    # 真实里 brain 就是 GPT
        if decision[0] == "final":
            return decision[1]                 # 给出最终答案
        _, tool, arg = decision
        result = TOOLS[tool](arg)              # 真的调用工具
        history.append(("observation", result))  # 结果回流,供下一轮决策
```

> 代码见 [`code/minimal_agent.py`](code/minimal_agent.py)
> (本例用规则模拟「大脑」以便稳定复现;真实场景把它换成 GPT 即可。)

配两样工具(计算器、知识库查询),跑两个任务:

```text
【问题】(12 + 8) * 3 等于多少?
  第1步 · 想:该用工具 [calculator]，参数「(12 + 8) * 3」
          · 看:工具返回「60」
  第2步 · 最终答案:答案是 60

【问题】北京人口的 2 倍大概是多少万?
  第1步 · 想:该用工具 [lookup]，参数「北京人口」
          · 看:工具返回「约 2184 万」
  第2步 · 想:该用工具 [calculator]，参数「2184 * 2」
          · 看:工具返回「4368」
  第3步 · 最终答案:北京人口约 2184 万,其 2 倍约为 4368 万
```

第二个任务尤其能说明问题:Agent **先「查」到人口、再「算」乘法**——
多步、多工具协作,这是单次问答根本做不到的。这正是 ChatGPT 的函数调用、
以及各类「AI 助手帮你订票查天气」背后的基本范式。

## 本章小结

- 模型有两个天生短板:**知识冻结**(不知道新事/私有文档)和**只会生成文字**(不会算术、上网、调 API)。
- **RAG**:回答前先检索资料库里最相关的几条,拼进提示词再生成。核心是「向量化 + 找最近邻」。补的是知识短板,还让答案有出处、少幻觉。
- **Agent**:给模型配工具,在 ReAct「想-做-看」循环里自己决定调哪个工具、看结果再决定下一步。补的是动手短板,能多步、多工具协作。
- 实跑印证:RAG 每问都精准命中对应资料;Agent 先查人口再算乘法,完成了单次问答做不到的多步任务。
- 真实系统更复杂(神经嵌入 + 向量库、GPT 当大脑、丰富的工具集),但内核就是本章这两套朴素骨架。

到这里,这套**从零理解大模型**的旅程就走完了:从一个字怎么变成向量,到注意力、Transformer、
训练出会说话的 GPT,再到采样、微调、对齐、推理优化,最后让它用上知识、用上工具。
你已经亲手把每一块拼图都摸过一遍——剩下的,就是拿这些零件去搭你自己的东西了。
