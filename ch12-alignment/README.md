# 第十二章 · 对齐入门(RLHF / DPO)

> 模型会说话了(第九章),还能调风格(第十章)、学新任务(第十一章)。但「会说」不等于「说得好」——
> 它可能啰嗦、跑题、甚至一本正经地胡说。怎么让它说得**有用、无害、合人意**?这一步叫**对齐(alignment)**。
>
> 这一章讲清对齐的主流思路:先看经典 **RLHF** 的第一步——从人类偏好里学一个**奖励模型**;
> 再看当下更简洁的 **DPO**,它把好几步压成一个损失,直接用偏好把模型「掰」过来。两个都手写、都能跑。

---

## 1. 难题:好不好,没有标准答案

前面每一章训练都靠一个明确的目标:下一个字是什么、改动量等于多少。但「这个回答好不好」呢?

- 太**主观**:同一个问题,一千个人有一千种「好回答」,写不出标准答案当监督信号。
- 可**比较**:但你给人看两个回答,问「哪个更好」,大多数人都能轻松选出来。

对齐就是抓住这一点:**不教标准答案,只收集「A 比 B 好」这种偏好**,再想办法把它变成训练信号。

---

## 2. RLHF 第一步:从偏好学出一把「尺子」(奖励模型)

经典 RLHF(基于人类反馈的强化学习)的第一步,是训练一个**奖励模型**:
给任意回答打一个分,好的高、差的低。它怎么从「只知道谁比谁好」里学出**具体分数**?

靠 **Bradley-Terry 模型**:假设每个回答有个隐藏质量,那么

```text
回答 a 胜过 b 的概率 = sigmoid(quality_a - quality_b)
```

质量差得越多,人越笃定;质量接近时,选择就接近抛硬币。反过来,只要有大量偏好对,
就能把每个回答的分数**反推**出来。损失就是这件事的负对数似然:

```python
# 逼着「赢家分 - 输家分」尽量大
loss = -log sigmoid(reward[赢家] - reward[输家])
```

> 代码见 [`code/reward_model.py`](code/reward_model.py)

我们造 5 个回答(各有一个我们假装不知道的「真实质量」),让人按 Bradley-Terry 给出 2000 条偏好,
然后只用偏好把分数学出来:

```text
  第   1 轮  loss = 0.6931
  第 100 轮  loss = 0.4110
  第 200 轮  loss = 0.3603
  第 400 轮  loss = 0.3361

===== 学到的奖励 vs 真实质量 =====
  完美的回答:  奖励模型打分 +1.95   (真实质量 +3.0)
  不错的回答:  奖励模型打分 +0.80   (真实质量 +1.5)
  还行的回答:  奖励模型打分 +0.02   (真实质量 +0.5)
  啰嗦的回答:  奖励模型打分 -0.84   (真实质量 -0.5)
  跑题的回答:  奖励模型打分 -1.93   (真实质量 -2.0)

奖励模型的排序是否和真实质量一致: True
```

学出的**绝对数值**和真实质量不完全相等(也不需要相等),但**相对高低、排序完全对上了**。
这正是奖励模型要的:当 RLHF 里那个会打分的「裁判」。

---

## 3. 经典 RLHF 的痛点

有了奖励模型,经典 RLHF 的下一步是:用强化学习(**PPO**)去推动语言模型——
多生成奖励高的回答。但这一步在工程上很「重」:

- 同时要摆弄**好几个模型**(策略模型、奖励模型、参考模型、价值模型),显存和复杂度都高。
- PPO **超参敏感、难调**,训练容易不稳定。

有没有办法,**跳过奖励模型、跳过 PPO**,直接拿偏好对把模型更新了?有——DPO。

---

## 4. DPO:把好几步压成一个损失

DPO(Direct Preference Optimization,直接偏好优化)的发现:RLHF 那两步,
**数学上可以合并成一步**。不训奖励模型,不跑 PPO,直接用偏好对更新模型本身。

它要同时做两件事,而损失天然就包含这两件:

1. **学偏好**:把「好回答 vs 坏回答」的概率差拉开。
2. **别跑偏**:所有概率都和一个**冻结的参考模型 `ref`**(对齐前的自己)比较,系数 `β` 当缰绳,
   防止模型为讨好而胡说、丢掉原有能力。

```python
# 每个回答相对 ref 的「提升」= policy 的 log 概率 - ref 的 log 概率
margin_win  = logp_policy[赢家] - logp_ref[赢家]
margin_lose = logp_policy[输家] - logp_ref[输家]
loss = -log sigmoid( β * (margin_win - margin_lose) )
```

> 代码见 [`code/dpo_train.py`](code/dpo_train.py)

为看清本质,这里把「模型」简化成:对同一问题的 5 个候选回答的一个概率分布。
策略从 `ref` 出发(初始一视同仁),只用偏好对训练:

```text
  第   1 步  loss = 0.6931
  第 100 步  loss = 0.3583
  第 300 步  loss = 0.3396
  第 500 步  loss = 0.3395

===== 对齐前后,模型选各回答的概率 =====
回答          真实质量       对齐前       对齐后
完美的回答       +3.0     0.165     0.947
不错的回答       +1.5     0.230     0.049
还行的回答       +0.5     0.133     0.003
啰嗦的回答       -0.5     0.193     0.001
跑题的回答       -2.0     0.279     0.000

对齐后,概率最高的回答是「完美的回答」(真实最优是「完美的回答」): True
```

对齐前,模型对 5 个回答几乎一视同仁(甚至给「跑题」最高概率 0.279);
**对齐后,概率被果断挪到了「完美的回答」(0.947),差回答被压到近 0**——
全程没训奖励模型、没跑 PPO,只用了一个损失。

---

## 5. 从「5 个回答的分布」到真正的 GPT

上面把回答简化成了一个 5 维分布,只为看清原理。真实场景里:

- 「每个回答的概率」就是语言模型对**整句回答**的生成概率(把每个字的 log 概率加起来);
- `policy` 和 `ref` 都是真正的 GPT,`ref` 是对齐前那一份的冻结副本;
- 偏好对来自人类标注:同一个 prompt,标注员选出更好的那条回答。

原理一模一样,只是把那个 5 维分布换成了一个真正的大模型。DPO 因为简单、稳定、好复现,
已经成为当下最流行的对齐方法之一。

## 本章小结

- **对齐**要解决的是「说得好不好」——它没有标准答案,但人类**擅长二选一比较**,这是一切的起点。
- **奖励模型**:用 Bradley-Terry 把一堆「A 比 B 好」的偏好,反推成一把能给任意回答打分的尺子。损失是 `-log sigmoid(好分 - 差分)`。
- **经典 RLHF**:奖励模型当裁判,再用 PPO 推模型多说高分回答;但要摆弄多个模型、超参敏感、难调。
- **DPO**:把「训奖励模型 + 跑 PPO」压成一个损失,直接用偏好对更新模型。处处和冻结的 `ref` 比较,`β` 当缰绳——既学会偏好,又不跑偏。
- 实跑印证:对齐后,好回答的概率被果断抬高、差回答压到近 0,而整体仍是合法分布。
- 真实场景只是把「K 个回答的分布」换成真正的 GPT,原理不变。DPO 因简单稳定,成了当下主流。
