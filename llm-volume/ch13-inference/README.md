# 第十三章 · 推理优化

> 模型训好(第九章)、对齐好(第十二章)了,真要拿来用,就得**一个字一个字地生成**。
> 这一步叫**推理(inference)**。它有两个绕不开的成本:**慢**(每步都在重算)和**大**(权重占满显存)。
>
> 这一章手写两个最经典的优化:**KV cache**——把算过的 K、V 存起来,别再重算;
> **int8 量化**——用更小的整数存权重,模型直接瘦一圈。两个都能跑,都能亲眼看到省了多少。

---

## 1. 慢在哪:自回归生成天生爱重算

回忆第六章,注意力要算 Q、K、V。生成时模型一个字一个字往外吐,每吐一个就要回看前面所有字:

```text
第 1 步:处理 "白"        -> 算 1 个字的 K、V
第 2 步:处理 "白日"      -> 把 "白" 的 K、V 又从头算一遍,再加 "日"
第 3 步:处理 "白日依"    -> "白日" 的 K、V 第三次重算……
```

每多生成一个字,前面所有字的 K、V 就被**整个重算一遍**。序列越长,浪费越夸张。

---

## 2. KV cache:算过的就存起来

主意特别朴素:**K、V 算过一次就缓存,下一步只算新来的那一个字**,把它的 K、V 追加进缓存。

```python
class KVCache:
    def step(self, new_token):
        q = new_token @ Wq
        k = new_token @ Wk            # 只算新字这一个 k
        v = new_token @ Wv            # 只算新字这一个 v
        self.K = vstack([self.K, k])  # 追加进缓存,不重算旧的
        self.V = vstack([self.V, v])
        scores = q @ self.K.T / sqrt(d)
        return softmax(scores) @ self.V
```

> 代码见 [`code/kv_cache.py`](code/kv_cache.py)

第一件要确认的事:**缓存不能改变结果**,它只是省计算。逐字生成,核对「每步重算」和「用缓存」:

```text
  第 1 步:序列长 1  两种方式输出一致 = True
  第 2 步:序列长 2  两种方式输出一致 = True
  ...
  第 6 步:序列长 6  两种方式输出一致 = True

全程输出完全一致: True  <- 缓存不改变结果,只省计算
```

省了多少?数一数两种方式各算了多少个字的 K、V:

```text
  每步重算(无缓存): 21 个   <- 1+2+3+...+T,随长度平方增长
  用 KV cache       : 6 个   <- 每步只算 1 个新字,线性增长
  省下              : 15 个字的重复计算
```

生成 6 个字就省了一大半;序列越长,差距越悬殊——计算量从**随长度平方增长**降到**随长度线性增长**。

代价是**显存**:得把所有历史 K、V 一直存着。这正是长上下文很吃显存的根本原因之一,
也是为什么几乎所有推理框架(vLLM、TensorRT-LLM……)都把 KV cache 当头号优化。

---

## 3. 大在哪:权重占满显存

KV cache 省的是计算。另一个成本是**存储**:大模型权重通常用 float32 存,**每个数 4 字节**。
一个 70 亿参数的模型,光权重就要 70 亿 × 4 ≈ 28 GB,普通显卡根本装不下。

能不能用更少的位数存每个数?能——**量化(quantization)**。

---

## 4. int8 量化:用 256 个台阶近似连续的数

int8 只能表示 -128~127 共 256 个整数。把 float32 权重压成 int8,**每个数从 4 字节缩到 1 字节,体积直接 1/4**。

关键技巧是**缩放(scale)**:找出这批权重里绝对值最大的数,把它映射到 127,其余按比例缩放取整;
用的时候再乘回 scale 还原。

```python
def quantize_int8(w):
    scale = abs(w).max() / 127.0          # 最大绝对值 -> 127
    q = round(w / scale).astype(int8)     # 按比例缩放、四舍五入到整数台阶
    return q, scale

def dequantize(q, scale):
    return q.astype(float32) * scale      # 乘回 scale,还原成近似 float
```

> 代码见 [`code/quantization.py`](code/quantization.py)

拿一个 256×256 的权重矩阵试,先看体积:

```text
  原始 float32:  262,144 字节  (每个数 4 字节)
  量化 int8   :   65,536 字节  (每个数 1 字节) + 1 个 scale
  压缩到原来的: 25.0%  <- 缩了 4 倍
```

代价是精度——int8 只有 256 个台阶,势必有四舍五入误差。但误差有多小?

```text
  权重相对误差: 0.0108  <- 有小幅四舍五入误差
  单个权重示例: 原始 +0.1257  ->  还原 +0.1118

  输出相对误差: 0.0100  <- 误差很小,模型行为基本不变
```

体积缩到 1/4,而权重和输出的相对误差都只有 1% 左右——**省一大半空间,几乎不影响行为**。这就是量化划算的地方。

## 本章小结

- **推理的两个成本**:慢(自回归每步都在重算)和大(权重占满显存)。
- **KV cache**:K、V 算过就存,下一步只算新字那一个,追加进缓存。结果完全一致,计算量从随长度平方降到线性。代价是要拿显存存历史 K、V。
- **int8 量化**:用 256 个整数台阶近似连续的 float,每个数从 4 字节压到 1 字节,体积缩到 1/4。靠 `scale`(最大绝对值映射到 127)缩放取整,用时乘回还原。
- 实测:量化后权重和输出的相对误差都约 1%,**省一大半空间却几乎不影响模型行为**。
- 真实做法更精细(分组量化、4-bit、KV cache 的分页管理等),但内核就是本章这两套朴素思想。
